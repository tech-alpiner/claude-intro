Claude in Action — QA Scenarios

What Does Claude Actually Do for QA?

Forget the tool names for a moment. Here is what QA engineers ask Claude every day — and what they get back.

🎫

Turn a ticket into a test plan

You say: "Here is the acceptance criteria for this feature. What test scenarios should I cover?"

Claude returns: happy-path tests, edge cases, boundary values, negative tests, and suggested manual verification steps.

🐛

Diagnose a failing test

You paste: the error message, the test code, the relevant function.

Claude returns: root cause hypothesis, the exact line causing the issue, and a suggested fix — all in under a minute.

🔍

Find what's not tested

You say: "Read this module and list the test coverage gaps."

Claude returns: a list of untested paths, missing edge cases, and which classes/functions have no test at all.

📋

Review a PR for regression risk

You say: "Here is the diff. What could break? What should I manually test?"

Claude returns: a structured risk summary, affected areas, and a checklist of regression scenarios to verify.

💡

All four scenarios work in Claude.ai (paste text), IDE extensions (select code), and Claude Code (full codebase). The prompt is the same — the power level scales with the tool.

Prompting — All Claude Tools

Writing Better Prompts

❌ Bad Prompt

fix the test
  • No file path given
  • No error message
  • No acceptance criteria
  • No verification step

✅ Good Prompt

The test at tests/test_orders.py::
test_update_status is failing:

AssertionError: expected 200 got 400

Read the test and the view at
com/myproject/monitoring/views.py.
Find the root cause. Fix it.
Run pytest and report pass/fail.

Prompt Template — Works in Claude.ai, IDE & Claude Code

Role:    You are a senior QA engineer on a Django/React web platform.
Goal:    [write tests / debug failure / analyze coverage gap]
Context: @[file paths] or paste code — constraints: [requirements]
Output:  [test file / bug report / edge case list]
Verify:  Run [pytest command] and report pass/fail with output.

Best Practices — All Claude Tools

Tips for Better Results

  • 🗂️ Always give file paths — use @path/to/file inline
  • ✅ Ask for verification every time — "run pytest and report results"
  • 🎯 Constrain scope — "only modify tests/test_orders.py"
  • 🔍 Ask for edge cases explicitly — "list 5 cases not covered"
  • 📋 Request structured output — "format as a markdown checklist"
  • 🔄 Explore → Plan → Implement — never skip the planning step for complex tasks
  • 🧹 Use /clear between unrelated tasks — resets context
  • 📏 Keep tasks small — one focused change is better than a giant rewrite
  • 🔒 Never paste secrets — no API keys, passwords, or tokens
  • 👀 Always review before committing — you own the output
The better your prompt and project context, the less Claude hallucinates. Context is everything.

Troubleshooting — All Claude Tools

When Claude Goes Off Track

🚨

Weird responses or errors?

Check status.anthropic.com first. If there's an active incident — just wait. No prompt will fix it.

🌐 Claude.ai & 🖥️ Claude Desktop

  • Start a new conversation — context builds up and confuses output
  • Check your Project Instructions are current and specific
  • Switch to a stronger model (Sonnet → Opus) for complex tasks
  • Restart the Desktop app if it feels slow
  • Sign out and back in on auth errors

🧩 IDE Extensions

  • Reload the IDE window: Ctrl+Shift+P → Reload Window
  • Re-sign in via the extension panel if auth expires
  • Update the extension — Claude extensions update frequently

📋 Universal Fixes

  • Be more specific — add file name, error text, expected outcome
  • Narrow scope — less context, more focused prompt
  • Start fresh after 5+ back-and-forth exchanges
  • Never paste API keys, passwords, or tokens

Claude.ai & Desktop — Projects

Claude Projects — Context That Persists

Create a Project and Claude remembers your stack, conventions, and role — across every conversation.

Setup in 5 steps

  1. Claude.ai or Desktop → New Project
  2. Name it: "MyProject QA"
  3. Write Project Instructions (see example →)
  4. Optionally upload reference docs (specs, test standards)
  5. Every chat in this project starts with full context

Why this matters

  • No more "explain our stack" on every session
  • Claude knows your test patterns from day one
  • Whole team can share the same project for consistent answers
  • Upload API specs, PRDs, or test checklists as knowledge

Example Project Instructions

# MyProject QA Project

## About This Project
Django/React web platform.
Backend: Python. Frontend: React.
Tests: pytest (backend), Playwright (E2E).

## Test Writing Rules
- All tests in com/myproject/*/tests/
- Use factories from tests/factories.py
- Happy-path + at least 3 edge cases per feature
- Never hardcode IDs in test data

## Test Commands
pytest com/myproject/tests/
npx playwright test

## Your Role
You are a senior QA engineer on this team.
Be specific, give real code examples, ask if unclear.

Claude Code — Configuration

CLAUDE.md — Your Project Context File

A markdown file in the root of your repo. Claude Code reads it first on every session — before it does anything else.

What to put in it

  • What the project is and what stack it uses
  • How to run tests, linters, and migrations
  • Team conventions (file structure, naming, patterns)
  • What Claude must never touch or modify

Create it automatically

cd your-project
claude
/init   ← Claude scans and drafts it

Keep it short, specific, and maintained.
A stale CLAUDE.md is worse than none — Claude follows outdated instructions confidently.

Example CLAUDE.md

# WandaHealth Platform

## Stack
Backend: Python / Django
Frontend: React / TypeScript
Tests: pytest, Jest, Playwright

## How to Run Tests
Unit:    pytest com/wanda/tests/
Frontend: npm run test
E2E:     npx playwright test

## QA Conventions
- All new features need happy-path + edge-case tests
- Test files live next to the modules they test
- Use factories from tests/factories.py
- Never hardcode IDs in test data

## Off-Limits
- Do not modify migrations directly
- Do not push directly to main

Claude Code — Skills

Skills — Custom Slash Commands for Your Team

Create a markdown file in .claude/commands/ — the filename becomes a /command available to everyone on the team.

How to create a skill

# .claude/commands/qa-review.md

You are a QA reviewer.
Read the diff of the current branch vs main.

List:
1. What changed (brief summary)
2. Regression risks
3. Test coverage gaps
4. Suggested manual test steps

Now any engineer types /qa-review and gets an instant structured QA review — no prompt copy-pasting.

Skills are committed to the repo — the whole team gets the same commands automatically, and they improve like any other code.

Useful skills to create

CommandWhat it does
/qa-reviewRead diff → list regression risks
/write-testsGenerate tests for the open file
/edge-casesList uncovered edge cases
/test-planProduce a test plan from a feature description
/regression-notesSummarise diff risks for a release

Hooks — run commands on events

# .claude/hooks/pre-commit.sh
pytest com/wanda/tests/ --tb=short -q

Hooks run automatically — e.g. run tests before every commit.

Claude Code — Safety

Restricting Commands — settings.json & settings.local.json

Control exactly what Claude Code can and cannot run. Deny list always beats allow list — no exceptions.

settings.json (committed — team-wide)

{
  "permissions": {
    "allow": [
      "Bash(pytest:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Bash(git status:*)",
      "Read(*)",
      "Write(tests/**)",
      "Edit(tests/**)"
    ],
    "deny": [
      "Bash(git push --force:*)",
      "Bash(git reset --hard:*)",
      "Bash(rm -rf:*)",
      "Bash(DROP TABLE:*)",
      "Bash(kubectl delete:*)"
    ]
  }
}

settings.local.json (gitignored — personal)

{
  "permissions": {
    "allow": [
      "Bash(pytest:*)",
      "Bash(git push:*)"
    ]
  }
}

File responsibilities

FileCommitted?Who it affects
settings.json✅ YesEveryone on the team
settings.local.json❌ GitignoredYou only

Always deny these

  • git push --force — overwrites teammates' work
  • rm -rf — irreversible deletion
  • DROP TABLE / DELETE FROM — DB destruction
  • kubectl delete — infrastructure removal
  • git reset --hard — discards uncommitted work
